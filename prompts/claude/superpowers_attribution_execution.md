# jDevFlow superpowers Attribution Prompt

아래 지시에 따라 `jDevFlow`의 `superpowers` 참고/차용에 대한 attribution 정리 작업을 실제 파일 수정까지 진행해줘.

중요:
- 이 작업은 법률 자문이 아니라, 공개 Git 저장소 공유를 위한 보수적 attribution 정리 작업이다.
- `superpowers`는 MIT License로 공개되어 있으므로, substantial portion으로 해석될 수 있는 차용이 있다면 보수적으로 출처/고지를 남기는 방향으로 정리해라.
- `superpowers/` 폴더는 참고만 하고 수정하지 마라.
- `jDevFlow`의 독자성을 유지하되, 영향을 받은 부분은 숨기지 마라.

먼저 읽을 파일:
- `docs/notes/2026-04-21-superpowers-attribution-handoff.md`
- `docs/notes/2026-04-21-superpowers-analysis-handoff.md`
- `README.md`
- `README.ko.md`
- `WORKFLOW.md`
- `.skills/README.md`
- 필요 시 이번 턴에서 추가/수정된 skill/template 문서
- 참고용: `superpowers/LICENSE`
- 참고용: `superpowers/README.md`
- 참고용: `superpowers/skills/test-driven-development/SKILL.md`
- 참고용: `superpowers/skills/writing-plans/SKILL.md`

이번 작업 목표:
1. `superpowers` 영향 범위를 정리
2. inspiration / adapted structure / close adaptation 수준을 구분
3. `README.md` acknowledgment 반영
4. 가능하면 `README.ko.md` 대응 문구 반영
5. 루트 `ATTRIBUTION.md` 작성

반드시 지켜야 할 원칙:
- 숨기지 말 것
- 과장하지 말 것
- 보수적으로 정리할 것
- jDevFlow의 독자성은 유지할 것

실행 순서:

## 1. 영향 범위 식별

먼저 아래를 짧게 정리해라.
- 어떤 부분이 inspiration only 인지
- 어떤 부분이 adapted structure 인지
- 어떤 부분이 close adaptation 으로 볼 수 있는지

## 2. attribution 전략 결정

다음을 결정해라.
- README에는 어떤 짧은 문구를 넣을지
- ATTRIBUTION.md에는 어떤 수준으로 설명할지
- 한국어 README에도 넣을지

기본 추천:
- README는 짧게
- ATTRIBUTION.md는 구체적으로

## 3. 실제 파일 수정

최소 변경:
- `README.md`
- `ATTRIBUTION.md`

권장 변경:
- `README.ko.md`

## 4. 최종 검토

아래를 점검해라.
- 충분히 성실한 attribution인지
- 실제보다 과장되지 않았는지
- 공개 Git 저장소에서 자연스럽게 읽히는지

## 5. 완료 후 응답 형식

최종 응답에는 반드시 아래를 포함해라.

1. 변경한 파일 목록
2. 어떤 수준의 영향을 attribution 대상으로 봤는지
3. README 문구를 어떻게 정했는지
4. ATTRIBUTION.md에 무엇을 넣었는지
5. 남은 불확실성

작업을 멈추지 말고, 실제 파일 수정과 정리까지 끝내라.
