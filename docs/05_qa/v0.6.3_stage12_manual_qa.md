## MQ-M1-1 — Monitor 재시작 후 신호 정상 캐치

**대상 파일:** `scripts/monitor_bridge.sh`, `scripts/setup_tmux_layout.sh`
**검증자:** 개발팀 (공기성 1.1 또는 현봉식 1.4)
**설계 근거:** design_final Sec.1.1 M1 / Sec.2.1 (F-62-7a/b/c)

### 사전 조건

- tmux 설치 및 `joneflow` 세션 활성화
- `bridge` 세션 존재 (`tmux new-session -d -s bridge` 또는 `setup_tmux_layout.sh` 실행)
- `scripts/monitor_bridge.sh` 빌드 완료 (≥30줄, 권한 `+x`)

### 실행 절차

```bash
# Step 1. bridge 세션 + Monitor 시작
export TMUX_SESSION_BRIDGE="bridge"
tmux new-session -d -s bridge 2>/dev/null || true
bash scripts/monitor_bridge.sh &
MONITOR_PID=$!
echo "Monitor PID: $MONITOR_PID"

# Step 2. 신호 발생 (bridge 세션에 test 신호 주입)
tmux send-keys -t bridge "echo '📡 status $(date +%Y-%m-%d\ %H:%M:%S) Stage 12 QA test signal ✅'" Enter
sleep 4

# Step 3. Monitor 강제 종료 후 재시작
kill $MONITOR_PID 2>/dev/null || true
sleep 1
bash scripts/monitor_bridge.sh &
MONITOR_PID2=$!
echo "재시작 Monitor PID: $MONITOR_PID2"

# Step 4. 재시작 후 신호 재발생
tmux send-keys -t bridge "echo '📡 status $(date +%Y-%m-%d\ %H:%M:%S) Stage 12 QA restart test ✅'" Enter
sleep 4

# Step 5. 정리
kill $MONITOR_PID2 2>/dev/null || true
```

### 판정 기준

| 항목 | PASS 조건 | FAIL 조건 |
|------|---------|---------|
| 초기 신호 캐치 | Monitor stdout에 `📡 status … Stage 12 QA test signal ✅` 출력 | 신호 미출력 또는 누락 |
| 재시작 후 신호 캐치 | 재시작 Monitor stdout에 `📡 status … restart test ✅` 출력 | 재시작 후 5초 내 신호 미캐치 |
| 중복 신호 없음 | 동일 신호 1회만 출력 (1초 dedup) | 동일 신호 2회 이상 연속 출력 |
| timestamp 형식 | `YYYY-MM-DD HH:MM:SS` 포함 | timestamp 누락 또는 형식 불일치 |

**최종 판정:** 4항목 모두 PASS → **PASS** / 1항목 이상 FAIL → **FAIL**

## MQ-M4-1 — tmux 1+3 pane 동시 spawn 안정성

**대상 파일:** `scripts/spawn_team.sh`, `scripts/setup_tmux_layout.sh`
**검증자:** 개발팀
**설계 근거:** design_final Sec.1.4 (M4-EP2) / Sec.6.4 (Q4) — Stage 8 mock 시뮬레이션

### 사전 조건

- tmux ≥1.8 설치
- `scripts/spawn_team.sh` 존재 (M4 drafter 산출물)
- 기존 `test-split` 세션 없음 (`tmux kill-session -t test-split 2>/dev/null || true`)

### 실행 절차 (mock 시뮬레이션)

```bash
# Step 1. 테스트 세션 생성 (4 pane 레이아웃)
tmux kill-session -t test-split 2>/dev/null || true
tmux new-session -d -s test-split -x 220 -y 50

# Step 2. 3개 pane 추가 (오케 1 + 팀원 3 구조)
tmux split-window -t test-split -h    # right
tmux split-window -t test-split:0.1 -v  # right-top / right-mid
tmux split-window -t test-split:0.2 -v  # right-mid / right-bot

# Step 3. 4개 pane 동시 신호 전송
for i in 0 1 2 3; do
  tmux send-keys -t "test-split:0.$i" "echo 'pane $i ready — $(date +%H:%M:%S)'" Enter
done
sleep 2

# Step 4. pane 응답 수집
for i in 0 1 2 3; do
  echo "=== pane $i ==="
  tmux capture-pane -t "test-split:0.$i" -p | tail -3
done

# Step 5. 동시 부하 테스트 (20초간 1초 간격 신호)
for round in $(seq 1 5); do
  for i in 0 1 2 3; do
    tmux send-keys -t "test-split:0.$i" "echo 'round $round pane $i'" Enter
  done
  sleep 1
done
sleep 1

# Step 6. 결과 수집 + 정리
echo "=== 최종 pane 상태 ==="
for i in 0 1 2 3; do
  echo "--- pane $i ---"
  tmux capture-pane -t "test-split:0.$i" -p | grep "round" | wc -l
done

tmux kill-session -t test-split
```

### 판정 기준

| 항목 | PASS 조건 | FAIL 조건 |
|------|---------|---------|
| 4개 pane 생성 | 레이아웃 오류 없이 4 pane 활성화 | pane 생성 실패 또는 레이아웃 붕괴 |
| 동시 신호 전달 | 4개 pane 모두 `pane N ready` 출력 | 1개 이상 응답 누락 |
| 부하 중 안정성 | 5 round × 4 pane = 20 신호, 누락 ≤ 2 | 누락 3 이상 또는 세션 crash |
| 입력 경합 없음 | 각 pane의 출력이 다른 pane 내용과 섞이지 않음 | pane 간 출력 혼입 |

**실패 시 fallback:** sequential spawn 권장 (안정성 우선, 효율 감소 허용).

## MQ-M4-2 — 리뷰어 conditional 환경 분기 동작

**대상 파일:** `scripts/ai_step.sh` (`ai_step_stage9_review_route`)
**검증자:** 개발팀
**설계 근거:** design_final Sec.2.1 (F-62-8) / Sec.3.4

### 사전 조건

- `.claude/settings.json`에 `stage9_review: "codex"` 존재 (현재 ✅)
- `scripts/ai_step.sh`에 `ai_step_stage9_review_route` 함수 존재 (현재 ✅)

### 실행 절차

```bash
# Case A: codex 미설치 환경 (현재 상태)
# command -v codex → 실패 → self-review fallback 기대
bash -c '
  source scripts/ai_step.sh 2>/dev/null || . scripts/ai_step.sh
  ai_step_stage9_review_route
'

# Case B: stage9_review 값 변경 (임시) — 기타 값 테스트
TMPFILE=$(mktemp)
cp .claude/settings.json "$TMPFILE"
python3 -c "
import json
with open('.claude/settings.json') as f: d = json.load(f)
d['stage_assignments']['stage9_review'] = 'claude'
with open('.claude/settings.json', 'w') as f: json.dump(d, f, indent=2, ensure_ascii=False)
"
bash -c '
  source scripts/ai_step.sh 2>/dev/null || . scripts/ai_step.sh
  ai_step_stage9_review_route
'
# 복구
cp "$TMPFILE" .claude/settings.json && rm "$TMPFILE"

# Case C: stage9_review=codex + codex CLI mock (PATH에 dummy 추가)
mkdir -p /tmp/mock_codex_bin
echo '#!/bin/sh' > /tmp/mock_codex_bin/codex
chmod +x /tmp/mock_codex_bin/codex
PATH="/tmp/mock_codex_bin:$PATH" bash -c '
  source scripts/ai_step.sh 2>/dev/null || . scripts/ai_step.sh
  ai_step_stage9_review_route
'
rm -rf /tmp/mock_codex_bin
```

### 판정 기준

| Case | 기대 출력 | PASS 조건 |
|------|---------|---------|
| A (codex 미설치) | `Stage 9: … codex CLI 미감지` + `self-review 모드` | 메시지에 "self-review" 또는 "미감지" 포함 |
| B (stage9_review=claude) | `Stage 9: Codex 환경 미감지. self-review 모드` | "self-review" 포함, "codex" 경로 비활성화 |
| C (codex 있음 + codex 설정) | `Stage 9: Codex review 경로 활성화` + `plugin-cc` 안내 | "Codex review 경로 활성화" 포함 |

**최종 판정:** 3 Case 모두 PASS → **PASS** / Case A 또는 B 실패 → **FAIL** (self-review fallback 누락)
