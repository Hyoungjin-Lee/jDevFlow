## Sec. 1. UX 플로우 (운영자 시나리오)

### 1.1 진입 시나리오

```
[운영자] /dashboard 슬래시 명령 입력
        │
        ▼
[Claude CLI] .claude/commands/dashboard.md → python3 scripts/dashboard.py 호출
        │
        ▼
[Python textual App] 부팅 + on_mount() (sync, F-D4)
        │
        ├─ Header 위젯 (PMStatusBar, F-D3 final)
        ├─ TeamRenderer 위젯 (박스 3개)
        ├─ PendingPushBox 위젯
        ├─ PendingQBox 위젯
        │
        ▼
[run_worker(thread=True)] 1초 polling 시작
        │
        ├─ PersonaDataCollector.fetch() → PersonaState[]
        ├─ PendingDataCollector.fetch() → PendingPush[] / PendingQuestion[]
        │
        ▼
[run_worker call_from_thread] UI 갱신 (sync)
        │
        ▼
[운영자 확인] 박스 3개 + PM status bar + Pending 박스 2개 시각 확인
        │
        ▼
[운영자] q (quit) 또는 Ctrl-C → on_unmount() → 종료
```

### 1.2 운영자 인터랙션 영역

| 동작 | 키 | 결과 |
|------|----|----|
| 진입 | `/dashboard` | textual App 시작 |
| 종료 | `q` 또는 Ctrl-C | App 종료 + 토큰량 dump (session-end.sh hook) |
| 갱신 강제 | `r` | 1초 polling 즉시 trigger |
| 도움말 | `?` | 단축키 모달 표시 |
| 다중 버전 fold | `f` | sub-row `└` 토글 (다중 bridge 동시 시) |

read-only 영역 영구 (의제 5 사고 9 충돌 회피, F-X-2 정합) — 모든 동작은 표시·제어만, 외부 write 0건입니다.

### 1.3 정보 우선순위 (행 시각 hierarchy)

운영자가 한 눈에 봐야 할 정보 우선순위는 다음과 같습니다.

1. **상태 아이콘** (`◉ working` / `○ idle` / `(idle)` placeholder) — 즉시 색상 인식
2. **페르소나 이름** (이름 + 직급 단축) — 18명 식별
3. **작업명** (working 시) — 현재 진행 중인 task 1줄 요약
4. **토큰량 + 진행률 바** (token 사용량 quantized 진행 시각) — 한도 인지
5. **스파크라인** (시간별 token 변동) — 추세 인지

| 영역 | 외부 margin | 내부 padding | 비고 |
|------|------------|-------------|------|
| Header (PMStatusBar) | bottom 1행 | left 1열 | 단일 행 |

| 행 내 컴포넌트 | — | label↔value 1열 | 단일 행 |
| Pending 박스 2개 | 사이 1열 | top·bottom 1행 / left·right 1열 | round border 포함 |

### 3.2 화면 너비 + 박스 분배

| 화면 너비 | 박스 분배 |
|---------|---------|
| ≥ 120 col (운영자 표준 Ghostty 와이드) | 박스 3개 가로 1행 (각 박스 너비 ≈ 40 col) |

## Sec. 4. 박스 layout (boundary #3 본문 결정)

### 4.1 border style

**채택: `border: round` (라운드 박스)**

근거:

1. Stage 5 Sec.8.2의 `border: solid` 잠정 정정 — `solid`는 직각 모서리로 시각 부담이 있고 다중 박스 동시 표시 시 단조롭습니다.
2. textual 0.50+에서 `border: round`는 표준 지원 + Unicode `┌─┐│└─┘` 호환.
3. 운영자 Ghostty (NerdFont 표준) 환경에서 라운드 모서리 `╭─╮│╰─╯` 정상 렌더 확인.
4. 박스 3개 + Pending 박스 2개 통일된 시안.

### 4.2 박스 헤더 형식

각 박스 상단 헤더는 다음 형식으로 박습니다.

```
╭──── 기획팀 (4명) ────╮
│ ◉ 박지영 PL  [작업명] [▓▓▓░░░░░] [▁▂▃▄] 64k │
│ ○ 김민교 RE              ─                   │
│ ○ 안영이 SR              ─                   │
│ ○ 장그래 JR              ─                   │
╰──────────────────────────────────────────────╯
```

| 영역 | 표기 |
|------|------|
| 헤더 | `╭── <팀명> (<인원>명) ──╮` 형식, 색상 = `$box_border` |
| 행 (working) | `◉ <이름> <직급> [작업명] [진행률 바] [스파크라인] <토큰량>` |
| 행 (idle) | `○ <이름> <직급>  ─` (dimmed, 작업명·진행률·스파크라인 모두 `─` 단일 dash) |
| 푸터 | `╰─` 헤더 너비 정합 동일 길이 |

### 4.3 박스 3개 매핑 + 인원

| 박스 | 팀명 | 인원 | 페르소나 (Stage 5 Sec.10.3 정합) |
|------|------|------|--------------------------------|
| 1 | 기획팀 | 4 | 박지영 PL / 김민교 RE / 안영이 SR / 장그래 JR |
| 2 | 디자인팀 | 4 | 우상호 PL / 이수지 RE / 오해원 SR / 장원영 JR |
| 3 | 개발팀 | 7 | 공기성 PL / 최우영 RE / 현봉식 SR / 카더가든 JR / 백강혁 RE / 김원훈 SR / 지예은 JR |

총 18명 = 4+4+7 = 15명? 아닙니다, 4+4+7 = **15명** + PM = 16명? 다시 확인:

- 기획팀 = 박지영 + 김민교 + 안영이 + 장그래 = 4명
- 디자인팀 = 우상호 + 이수지 + 오해원 + 장원영 = 4명
- 개발팀 = 공기성 + 최우영 + 현봉식 + 카더가든 + 백강혁 + 김원훈 + 지예은 = 7명
- 합계 = 4 + 4 + 7 = **15명**

CTO 백현진 + PM 스티브 리 + CEO 이형진 = 3명 추가 시 18명. **F-D3 final 19명 표시 = 18명 박스 + PM 1명 = 19명**의 "18명"은 CTO 백현진 + PM 스티브 리 빼고 4팀(기획4 + 디자인4 + 개발7) + ?. 

**finalizer 영역에서 정정 확인 필요 (R-N 마커 권고):** Stage 5 Sec.10.3 본문 또는 `personas_18.md` 의존 영역. drafter는 18명 = 4+4+7 = 15명 + 추가 3명 영역으로 표시. 슬롯 인터페이스만 박는 본 stage 책임 영역에서는 **15명 박스 슬롯 + 3명 추가 슬롯 (Stage 5 Sec.10.3 또는 personas_18.md 영역)** 형식으로 본문 박음. detail Stage 8 영역.

| 컴포넌트 | 너비 (col) | 색상 변수 | 설명 |
|---------|-----------|---------|------|
| 상태 아이콘 | 1 | `$persona_working` (◉) / `$persona_idle` (○) | working / idle |
| 이름 | 8 (한국어 4자 = 8 col) | default | 페르소나 이름 |
| 직급 | 4 | dimmed | PL / RE / SR / JR |
| 작업명 | 12 (truncate `...`) | default | 현재 task 1줄 요약 |
| 진행률 바 | 8 | `$progress_bar_fill` / `$progress_bar_track` | token 사용량 quantized |
| 스파크라인 | 8 | `$sparkline_high` ↔ `$sparkline_low` | 시간별 token 변동 |
| 토큰량 | 6 (오른쪽 정렬) | default | `64k` / `128k` 형식 (k 단위) |

총 너비 = 1 + 8 + 4 + 12 + 8 + 8 + 6 + spacing 5 = **52 col** (박스 1개 너비 ≈ 40~50 col 정합).

### 5.2 idle 행 spec

```
○ <이름 8> <직급 4>  ─
```

idle 행은 작업명·진행률·스파크라인·토큰량 모두 single `─` dimmed dash로 압축합니다 (boundary #6 정합 — `(idle)` placeholder 행 단위 적용).

### 5.3 다중 버전 sub-row 표시

다중 bridge 동시 시(예: bridge-063 + bridge-064 동시 가동), 동일 페르소나 행 아래 `└` sub-row로 1행 표시합니다.

```
◉ 박지영 PL [v0.6.4 plan] [▓▓▓░░] [▁▃▂] 32k
└ ◉ 박지영 PL [v0.6.5 plan] [▓░░░░] [▁▁▁] 8k
○ 김민교 RE  ─
...
```

`MAX_SUB_ROWS = 1` (Stage 5 Sec.8.3 정합). 3개 이상 동시 가동 시 가장 최근 1개만 표시 + 나머지는 `+N more` 하단 표시.

## Sec. 6. 진행률 바 (boundary #4 본문 결정)

### 6.1 8칸 quantization spec

진행률 바는 8칸 가로 + 4단계 quantization으로 박습니다.

| 진행률 (%) | 표시 |
|-----------|------|
| 0~12.5 | `░░░░░░░░` |
| 12.5~25 | `█░░░░░░░` |
| 25~37.5 | `█▓░░░░░░` |
| 37.5~50 | `███░░░░░` |
| 50~62.5 | `████░░░░` |
| 62.5~75 | `█████░░░` |
| 75~87.5 | `██████▓░` |
| 87.5~100 | `███████▓` |
| 100 | `████████` |

quantization 단계 = `█` (full) / `▓` (3/4) / `▒` (1/2) / `░` (1/4 또는 empty).

### 6.2 적용 영역

| 적용 영역 | 분모 | 분자 |
|----------|------|------|
| token 사용량 | 모델별 한도 (Opus 1M = 1,000,000 / Sonnet 200k = 200,000 / Haiku 200k = 200,000) | 현재 누적 토큰 |
| thinking 진행도 | Stage 별 평균 thinking 토큰 (모델 등급별 평균 학습값, Stage 8 영역) | 현재 thinking 토큰 |

본 v1은 token 사용량 진행률 바를 기본으로 박고, thinking 진행도는 Stage 8 구현 시 토글 옵션으로 추가 가능 영역입니다.

### 6.3 색상 + 임계값

| 임계값 | 색상 | 의미 |
|-------|------|------|
| 0~70% | `$progress_bar_fill` (#58a6ff blue) | 정상 |
| 70~90% | `$pending_push_accent` (#d29922 yellow) | 경고 |
| 90~100% | `$pending_q_accent` (#f85149 red) | 한도 임박 |

임계값 적용은 `class` 동적 swap으로 처리 (textual reactive 표준).

## Sec. 7. 스파크라인 (boundary #5 본문 결정)

### 7.1 8칸 + 8단계 quantization spec

스파크라인은 8칸 가로 + 8단계 quantization으로 시간별 token 변동을 박습니다.

| 단계 | 문자 | 데이터 영역 (분모 = 시간 윈도우 max) |
|------|------|----------------------------------|
| 0 (최저) | `▁` | 0~12.5% |
| 1 | `▂` | 12.5~25% |
| 2 | `▃` | 25~37.5% |
| 3 | `▄` | 37.5~50% |
| 4 | `▅` | 50~62.5% |
| 5 | `▆` | 62.5~75% |
| 6 | `▇` | 75~87.5% |
| 7 (최고) | `█` | 87.5~100% |

### 7.2 시간 윈도우 + 데이터 buffer

| 영역 | 값 |
|------|----|
| 시간 윈도우 | 최근 8 polling cycle (8초, polling 1초 정합) |
| 데이터 buffer | `collections.deque(maxlen=8)` per persona |
| 분모 | 윈도우 내 max(token_delta) (정규화) |
| 분자 | 각 cycle의 token_delta |

### 7.3 색상 그라데이션

```css
.sparkline_char_low { color: $sparkline_low; }    /* ▁▂▃ */
.sparkline_char_mid { color: $progress_bar_fill; } /* ▄▅ */
.sparkline_char_high { color: $sparkline_high; }   /* ▆▇█ */
```

3단계 색상 그라데이션 = 안정/중간/급증 시각 인지 향상.

### 7.4 표시 위치

행 우측 끝, 토큰량 직전 (Sec.5.1 정합):

```
... [▓▓▓░░░░░] [▁▂▃▄▅▆▇█] 64k
```

## Sec. 8. 빈 박스 placeholder (boundary #6 본문 결정)

### 8.1 idle 시 행 표시

페르소나 1명 단위 idle은 Sec.5.2 정합 — `○ <이름> <직급>  ─` 단일 dash 압축.

### 8.2 박스 전체 idle 시

박스 안 모든 페르소나가 idle인 경우 (예: 디자인팀 4명 모두 idle), 박스는 다음 형식으로 박습니다.

```
╭──── 디자인팀 (4명) ────╮
│  (idle)                │
│                        │
╰────────────────────────╯
```

| 영역 | 표기 |
|------|------|
| 박스 헤더 | 정상 표기 유지 |
| 박스 본문 | `(idle)` 회색 dimmed 1행 + bottom 1행 padding |
| 박스 border | `$box_border` dimmed (alpha 0.5 또는 어두운 색) |

### 8.3 tmux 미존재 시 (no active session)

박스 전체에 해당 팀 tmux 세션이 0개인 경우 (예: bridge-* 0개), 다음 형식으로 박습니다.

```
╭──── 디자인팀 (4명) ────╮
│  no active session     │
│                        │
╰────────────────────────╯
```

`(idle)` vs `no active session` 구분 = `last_update` 보존 여부(F-D1 R-1 정합) — `last_update` 존재 시 `(idle)` / 없으면 `no active session`.

### 8.4 Pending 박스 빈 상태

Pending Push 박스 / Pending Q 박스도 빈 상태 placeholder를 박습니다.

```
╭──── Pending Push (3줄) ────╮
│  no pending push           │
╰────────────────────────────╯
```

```
╭──── Pending Q ────╮
│  no pending q     │
╰───────────────────╯
```

### 10.4 다중 bridge 통합 인입 (R-8 정합)

`PMStatusBar._fetch_pm_state()`는 `tmux ls`에서 `bridge-*` 패턴 모두 통합 인입합니다.

```python
# scripts/dashboard/status_bar.py (Stage 8 영역, 본 stage = 슬롯만)
def _fetch_pm_state() -> PMStatusBarState:
    """다중 bridge 동시 시 통합 표기. R-8 정합."""
    bridges = subprocess.check_output(["tmux", "ls"]).decode()
    bridge_names = [line.split(":")[0] for line in bridges.splitlines() if line.startswith("bridge-")]
    return PMStatusBarState(
        bridges=bridge_names,  # ["bridge-063", "bridge-064"]
        status="working" if bridge_names else "idle",
        ...
    )
```
