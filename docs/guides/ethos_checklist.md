# ETHOS 자체 점검 목록 (v0.6.3 M3)

> **목적:** Stage 진행 전·후 자체 점검. 경고 모드 — 항목 미충족 시 진행 차단 금지, 오케에게 알림 후 자율 수정.

---

## 1. Boil the Lake — 범위 경계 점검

"호수 전체를 끓이려 하지 않는다" — 현재 Stage 범위 내 작업만 수행.

**점검 항목:**

- [ ] 현재 Stage의 설계 문서(design_final.md) 또는 dispatch 브리프를 읽고 범위를 확인했는가?
- [ ] Stage 범위 밖의 마일스톤·파일을 수정하고 있지 않은가?
- [ ] "이왕 하는 김에 X도 고치자"는 충동을 억제했는가?
- [ ] 변경 대상 파일 목록(Sec.9 신규/개정)에 없는 파일을 건드리고 있지 않은가?
- [ ] 운영자 결정 영역 발생 시 즉시 보고하고 독단 결정하지 않았는가?

**트리거:** Stage 시작 전, 파일 수 5개 초과 수정 시.

---

## 2. autoplan — 단계별 절차 점검

"행동 전 계획, 계획 전 읽기" — 작업 전 브리프와 설계 문서 확인 후 착수.

**점검 항목:**

- [ ] 본 Stage의 dispatch 브리프를 실제로 읽었는가?
- [ ] design_final.md(또는 plan_final.md) 해당 마일스톤 섹션을 확인했는가?
- [ ] 작업 순서를 의존 그래프(M1→M3→M4→M5, M2 독립) 기준으로 정했는가?
- [ ] 위험 명령 실행 전 --dry-run 또는 echo 출력으로 사전 확인했는가?
- [ ] AC 검증 명령을 작업 완료 후 직접 실행할 계획이 있는가?

**트리거:** 새 작업 착수 전, 구현 단계 진입 전.

---

## 3. /investigate — 탐색 모드 점검

"모르면 먼저 읽어라" — 기존 코드·문서 파악 없이 새 코드 작성 금지.

**점검 항목:**

- [ ] 수정 대상 파일의 현재 내용을 실제로 읽었는가?
- [ ] 기존 함수·변수 명명 규칙을 파악하고 일관성을 유지했는가?
- [ ] 동일 기능이 이미 구현돼 있지 않은지 `grep`으로 확인했는가?
- [ ] 설계 제약(F-62-x)과 상충되는 구현을 하고 있지 않은가?
- [ ] 불명확한 사항은 추천 자동 적용(자율 모드) 또는 오케 보고로 해소했는가?

**트리거:** 새 파일 생성 전, 기존 파일 대규모 수정 전.
