### 8.1 R-11 (Critical 보강 권고) — Sec.9.3 `_dedupe_check` `lru_cache` 패턴 dedupe 미작동 영역

drafter Sec.9.3:
```python
class OSAScriptNotifier(Notifier):
    """macOS 기본 — Pushover / CCNotify 회피 (Q3 운영자 결정)."""

    @lru_cache(maxsize=128)
    def _dedupe_check(self, dedupe_key: str) -> bool:
        """5분 TTL — 동일 dedupe_key 중복 발화 방지 (F-X-7-#3)."""
        return True

    def notify(self, q: PendingQuestion) -> bool:
        if not self._dedupe_check(q.dedupe_key()):
            return False  # 중복 — skip
        ...
```

위 패턴은 `lru_cache`를 dedupe 도구로 사용했는데 **lru_cache는 메모이제이션 도구**이고 dedupe 도구가 아닙니다. 동작 trace:
1. 첫 호출 `_dedupe_check("Q1:12:34:00")` → 캐시 미스 → 함수 실행 → `True` 반환 → 캐시에 (`"Q1:12:34:00"` → `True`) 저장
2. `notify()`에서 `if not True:` → `False` 분기 미진입 → notify 발화 ✓ (정상)
3. 두 번째 호출 `_dedupe_check("Q1:12:34:00")` → 캐시 히트 → `True` 반환 (캐시된 값)
4. `notify()`에서 `if not True:` → `False` 분기 미진입 → **notify 다시 발화 ✗ (dedupe 안 됨)**

즉 `_dedupe_check`는 같은 `dedupe_key`로 호출되어도 항상 `True` 반환 → **dedupe 영구 미작동 영역**으로 보입니다. drafter Sec.9.3 의도는 "첫 호출 = `True` 발화 / 재호출 = `False` skip"인데 `lru_cache` 메모이제이션 패턴은 항상 `True` 반환이라 의도 미달.

올바른 dedupe 패턴:
```python
class OSAScriptNotifier(Notifier):
    def __init__(self):
        self._sent_keys: dict[str, datetime] = {}  # key → 첫 발화 시각

    def _is_recently_sent(self, dedupe_key: str, ttl_minutes: int = 5) -> bool:
        now = datetime.now()
        last_sent = self._sent_keys.get(dedupe_key)
        if last_sent and (now - last_sent).total_seconds() < ttl_minutes * 60:
            return True  # 최근 발화 → dedupe (skip)
        self._sent_keys[dedupe_key] = now
        return False  # 신규 발화

    def notify(self, q: PendingQuestion) -> bool:
        if self._is_recently_sent(q.dedupe_key()):
            return False  # 중복 — skip
        ...
```

drafter Sec.9.4 본문:
> `PendingQuestion.dedupe_key()`가 timestamp를 분 단위로 안정화하므로(Sec.3.2), `lru_cache(maxsize=128)`은 자연스럽게 5분 단위 dedupe를 구현합니다.

위 본문은 잘못된 영역으로 보입니다. `lru_cache`는 시간 기반 expiry / TTL 영역이 아니라 LRU eviction(최대 128개 키 누적 후 가장 오래된 키 제거) 영역. **분 단위 안정화된 키가 LRU 캐시에 누적되면 같은 키 재호출 시 항상 캐시 히트 → 함수 본체 미실행 → dedupe 의도와 정반대 동작.**

drafter Sec.9.4 마지막 줄:
> 정확한 TTL 만료(시간 기반 expiry)가 필요하면 Stage 8에서 `cachetools.TTLCache(maxsize=128, ttl=300)` 교체 영역으로 보입니다(Stage 8 이월 AC-S8-4).

`cachetools.TTLCache` 교체는 Stage 8 영역으로 박혔지만, **본 stage drafter v1 의사 코드 자체가 dedupe 미작동**이라 Stage 8 진입 직전까지 본 v1을 따르면 알림이 매 polling마다 발화하여 운영자 알림 폭주 가능 영역으로 보입니다(brainstorm 의제 6 알림 영역 영향).

**보강 권고 (Critical):** 다음 3옵션 중 finalizer 1건 결정 박음 권고:

1. **본 stage에서 정정 박음** — `_dedupe_check`를 `_is_recently_sent` (`dict[str, datetime]` + TTL 비교) 패턴으로 정정 + `dedupe_key()` 5분 단위 truncate(R-4 보강과 한 묶음). 본 stage closure 가능 영역.

## Sec. 11. 본 review가 다루지 않는 것

본 review는 **reviewer 영역 책임 범위**에 한정합니다. 다음 영역은 reviewer 권한 밖입니다.

- **Score 가중치 분배 + 확정 점수 박음** — finalizer (현봉식) 영역. drafter Sec.18 표 + 본 review R-12(Score 침범 권고)는 finalizer 자율 결정 영역.
- **AC verbatim 마커 위치 정정** — finalizer 자율 영역. 본 review는 AC 정합성 점검(R-5, R-9)만 박음, 정정 본문은 finalizer 영역.
- **commit msg 표현 + commit 분리 단위** — dispatch 자율 영역(회의창 작성 영역 X). 본 review 영역 외.
- **Stage 8 구현 detail (코드 본문 / 단위 테스트 본문)** — Stage 8 구현 영역. 본 review는 Stage 5 design 영역만.
- **textual cross-platform 실측 검증** — Stage 8/11 영역. drafter Sec.10.1 platform_compat detect_platform 본문은 자율 영역, 실측은 Stage 8 영역.
- **claude CLI hook API 환경변수 / stdin spec 검증** — Stage 8 영역(AC-S8-2). drafter Sec.7.3 의사 코드는 자율 영역, 실측은 Stage 8 영역.
- **dispatch 본문 작성 (디자인팀 첫 등판)** — 회의창(CTO 실장) + 박지영 PL 자율 영역. 본 review 영역 외.
- **운영자 결정 게이트 (DEFCON / Score < 80% / AC 자동 < 70% / 헌법 위반 / 사고 12·13 재발)** — dispatch 중단 조건 0건 발동(본 review 결과 정합)이라 본 review의 운영자 보고 0건 영역. R-10 글로벌 영역 침범 가능성은 finalizer 흡수 영역 또는 운영자 결정 게이트 후보 영역으로 분류 가능 영역(finalizer 자율 결정).
- **글로벌 ~/.claude/CLAUDE.md 영역** — F-62-4 v0.6.3 이월 영역. 본 v0.6.4 scope 외 영역(R-10 글로벌 영역 침범 가능성은 본 v0.6.4 영향 검토 영역).
