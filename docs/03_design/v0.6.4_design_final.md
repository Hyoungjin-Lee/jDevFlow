### 3.2 PendingPush / PendingQuestion dataclass 본문 (drafter v1 verbatim + R-4 정정)

```python
@dataclass(eq=True, frozen=False)
class PendingPush:
    """M4 — 운영자 승인 대기 push/commit (read-only 표시 only)."""

    item_id: str           # "push-001", "commit-002"
    item_type: Literal["push", "commit"]
    description: str       # "v0.6.4 Stage 2 planning commit"
    timestamp: datetime
    initiator: str         # 페르소나 이름
    severity: Literal["high", "medium", "low"]


@dataclass(eq=True, frozen=False)
class PendingQuestion:
    """M4 — 운영자 결정 영역 큐 (read-only 표시, 응답은 회의창 흐름 유지)."""

    q_id: str              # "Q1", "Q5", "Q-NEW-1"
    category: Literal["scope", "decision", "risk", "approval"]
    description: str
    source: str            # "dispatch/..._planning.md"
    timestamp: datetime
    priority: Literal["critical", "high", "medium", "low"]

    def dedupe_key(self) -> str:
        """알림 dedupe 5분 TTL용 해시 키 (Q3 / F-X-7-#3 흡수, R-4 정정 = 5분 단위 truncate)."""
        # R-4 정정: timestamp를 5분 단위로 truncate (drafter v1 분 단위 → 5분 단위)
        minute = self.timestamp.minute - (self.timestamp.minute % 5)
        truncated = self.timestamp.replace(minute=minute, second=0, microsecond=0)
        return f"{self.q_id}:{truncated.isoformat()}"
```

**finalizer 정정 (R-4):** drafter v1 `replace(second=0, microsecond=0)` (분 단위 = 1분 dedupe) → `minute - (minute % 5)` (5분 단위 truncate)로 정정. 운영자 결정 인수 Q3 + planning_review F-X-7 #3 "5분 TTL" 정합. R-11 Critical 정정과 한 묶음으로 dedupe 의도 작동 보장.

`dedupe_key()` 메서드는 5분 단위로 timestamp를 truncate하여 같은 5분 윈도우 내 동일 Q는 동일 키 생성 → `Notifier._is_recently_sent()`(R-11 정정 후) 측 `dict[str, datetime] + 5분 TTL` 비교 패턴과 정합합니다(Sec.9.3).

## Sec. 5. F-D4 본문 결정 — Threaded sync wrapper 채택

### 5.1 채택 결정 (drafter v1 verbatim)

본 v0.6.4 데이터 수집·렌더링 인터페이스는 **Threaded sync wrapper** 방식을 채택합니다.

- 모든 `Collector` 메서드는 sync `def`로 정의합니다 (`async def` 0건).
- `DashboardApp.on_mount()`에서 `self.run_worker(self._refresh_loop, thread=True)`로 백그라운드 sync 워커 실행.
- 워커가 1초마다 (Sec.7.4 갱신 주기) `PersonaDataCollector.fetch_all_personas()` + `PendingDataCollector.get_pending_*()`를 sync 호출 → `self.call_from_thread(self._update_widgets, ...)`로 메인 thread에 결과 push.

### 5.2 채택 근거 (drafter v1 verbatim)

| 옵션 | 장점 | 단점 | 채택 여부 |
|------|------|------|---------|
| **A. async/await 전면 채택** | textual 권장 패턴 일치 / I/O blocking 회피 | tmux subprocess는 sync 호출이라 `asyncio.to_thread` 필수 / 학습 비용 / Stage 8 구현 +6h | ❌ 회수 |
| **B. sync 메인 thread blocking** | 가장 단순 | textual 이벤트 루프 정지 (1~2초 freeze) — 사용자 가시 정지 발생 | ❌ 회수 |
| **C. Threaded sync wrapper (채택)** | sync 인터페이스 유지 / 메인 thread 비차단 / textual `run_worker(thread=True)` 표준 패턴 / Stage 8 구현 +0h | thread-safety 주의 (call_from_thread 박음) | ✅ 채택 |

**채택 근거 핵심 3건 (drafter v1 verbatim):**
1. M2 `PersonaDataCollector` / M4 `PendingDataCollector`의 외부 의존(tmux subprocess / file IO)은 **본질적으로 sync**입니다. `subprocess.run()` / `open().read()`는 `asyncio.to_thread` 래핑이 필요해서 옵션 A는 실질적으로 옵션 C와 동일한 thread 사용을 강요합니다. 인터페이스만 async로 바뀔 뿐 실행 모델은 동일합니다.
2. textual 0.50+에서 `App.run_worker(thread=True)`는 표준 패턴으로 안정화되어 있으며, Stage 4 plan_final F-D4 stub("sync 시작 / async 채택은 Stage 5")의 자연스러운 계승입니다.
3. M1 `on_mount()` 동기 훅과 정합성이 그대로 유지됩니다. M2 v1의 `persona_by_name`만 동기였던 일관성 깨짐은 sync 전면 통일로 해소됩니다.

### 5.3 thread-safety 영구 정책 (Stage 8 강제, R-3 정정 후)

```python
# scripts/dashboard.py (finalizer 정정 후 — R-3 sync 통일)

import time
import threading
from textual.app import App
from textual.binding import Binding
from scripts.dashboard.persona_collector import PersonaDataCollector
from scripts.dashboard.pending_collector import PendingDataCollector
from scripts.dashboard.notifier import OSAScriptNotifier


class DashboardApp(App):
    BINDINGS = [Binding("q", "quit", "Quit")]

    def on_mount(self) -> None:  # R-3 정정: async def → def (sync 통일)
        self.persona_collector = PersonaDataCollector()
        self.pending_collector = PendingDataCollector()
        self.notifier = OSAScriptNotifier()
        self._exit_signal = threading.Event()
        self.run_worker(self._refresh_loop, thread=True, exclusive=True)

    def _refresh_loop(self) -> None:
        """Worker thread — 1초 polling sync."""
        while not self._exit_signal.is_set():
            try:
                personas = self.persona_collector.fetch_all_personas()
                pushes = self.pending_collector.get_pending_pushes()
                questions = self.pending_collector.get_pending_questions()
            except Exception as exc:
                # Sec.14 에러 경로 — staleness 표시로 폴백
                self.call_from_thread(self._show_stale, exc)
            else:
                self.call_from_thread(self._update_widgets, personas, pushes, questions)
            time.sleep(1.0)
```

**finalizer 정정 (R-3):** drafter v1 `async def on_mount` → `def on_mount` (sync) 통일. textual `App.on_mount`는 sync `def`도 지원하며 `run_worker(thread=True)` 호출은 sync 컨텍스트에서도 정상 작동(textual 0.50+ 표준). F-D4 본문 결정 "sync 통일 + async 0건" 정합. `_exit_signal = threading.Event()` 초기화도 명시.

`call_from_thread` 박음이 thread-safety 영구 정책입니다. Worker thread에서 textual 위젯을 직접 변경하면 race condition이 발생할 수 있어 Stage 9 코드 리뷰 자동 검증 영역(Sec.15.4)에 포함합니다.

### 7.2 상태 추론 알고리즘 (Q5 idle 통합 confirm 본문, R-1 정정)

```python
def _infer_state(self, name: str, team: str, sessions: List[str]) -> PersonaState:
    """Q5 idle 통합 confirm — tmux 세션 미존재 = idle.

    절차:
        1. 페르소나 ↔ tmux session 매핑 조회 (drafter 자율).
        2. 매핑 세션이 active_sessions 리스트에 없음 → status = idle, task = None.
           **R-1 정정: last_update는 이전 값 보존 (없으면 epoch 0 = stale 표시 자연 발생).**
        3. 있으면 capture-pane 마지막 N초 변화 비교 → working/idle.
        4. working이면 task 추론 (A-1 prompt > A-3 로그 > A-2 thinking, F-X-7-#5).
           **drafter R-2 정정: A-2 thinking = last_known_task 직전 작업명 유지 채택.**
        5. tokens_k = TokenHook 인입 또는 capture-pane regex fallback.
    """
    pane_name = self._persona_to_pane(name)  # 예: "박지영" → "Orc-064-plan:1.1"
    if not pane_name or pane_name.split(":")[0] not in sessions:
        # R-1 정정: last_update를 이전 값(_last_known_states)에서 보존
        prev = self._last_known_states.get(name)
        last_update = prev.last_update if prev else datetime.fromtimestamp(0)
        return PersonaState(
            name=name, team=team, status="idle", task=None,
            tokens_k=0.0, last_update=last_update,  # R-1: datetime.now() → 보존값
        )

    last_change = self.tmux.last_pane_change(pane_name)
    elapsed = (datetime.now() - last_change).total_seconds()
    if elapsed > 10.0:  # T 임계 10초 (Stage 8 settings.json 가능)
        return PersonaState(
            name=name, team=team, status="idle", task=None,
            tokens_k=self.token_hook.get_tokens_k(pane_name),
            last_update=last_change,  # R-1: tmux 마지막 pane 변화 시각 보존
        )

    return PersonaState(
        name=name, team=team, status="working",
        task=self._infer_task(pane_name),  # A-1 > A-3 > A-2 (drafter R-2 정정 후 last_known)
        tokens_k=self.token_hook.get_tokens_k(pane_name),
        last_update=last_change,
    )
```

**finalizer 정정 (R-1 / drafter R-1·R-4 통합):** drafter v1 idle 폴백 `last_update=datetime.now()` → `last_update = prev.last_update or epoch 0`로 정정. staleness ⚠ 표시(Sec.14.3, `last_update > 2초` → `⚠`)가 정상 작동 — tmux 세션이 정말 오래 idle한 페르소나는 `last_update`가 갱신되지 않아 자연스럽게 stale 표시 발동. 운영자 가시성 신규 정책 정합 영역.

### 7.5 작업명 추론 우선순위 (F-X-7-#5 본문 흡수, drafter R-2 정정)

```python
def _infer_task(self, pane_name: str) -> Optional[str]:
    """A-1 prompt > A-3 로그 > A-2 last_known_task — F-X-7-#5 + drafter R-2 정정 흡수."""
    last_lines = self.tmux.capture_pane(pane_name, lines=50)

    # A-1: prompt 행 ">" 이후 텍스트
    for line in reversed(last_lines):
        m = re.search(r"^\s*>\s*(.{5,80})", line)
        if m:
            return self._extract_task_from_prompt(m.group(1))

    # A-3: 로그 라인 — "S\d+" / "Stage \d+" / "v\d+\.\d+"
    for line in reversed(last_lines):
        m = re.search(r"(v\d+\.\d+\.\d+[^\s]*|Stage \d+|S\d+ [^\s]+)", line)
        if m:
            return m.group(1)

    # A-2: drafter R-2 정정 — last_known_task 직전 작업명 유지 (None 회수)
    pane_session_pane = pane_name  # "Orc-064-dev:1.2"
    persona_name = self._pane_to_persona(pane_session_pane)
    prev = self._last_known_states.get(persona_name) if persona_name else None
    return prev.task if prev and prev.task else None
```

```python
    EMPTY_BOX_LABEL = "(팀 대기 중)"  # F-X-7-#6 옵션 A 흡수
    SUB_ROW_PREFIX = "└"               # F-M3-4 흡수
    TOKEN_FORMAT = "{:.1f}k"           # F-M3-3 흡수

    def render(self, teams_data: Dict[str, List[PersonaState]]) -> str:
        """드래프터 자율 — Vertical(Static * N) 구조 반환."""
        rendered = []
        for team in ["기획", "디자인", "개발"]:
            personas = teams_data.get(team, [])
            rendered.append(self._render_box(team, personas))
        return "\n".join(rendered)

    def _render_box(self, team: str, personas: List[PersonaState]) -> str:
        if not personas:
            return f"╔══ {team}팀 ══╗\n║ {self.EMPTY_BOX_LABEL} ║\n╚════════╝"

        grouped = self._group_by_name(personas)  # {"박지영": [PersonaState, PersonaState]}
        lines = [f"╔══ {team}팀 ══╗"]
        for name, states in grouped.items():
            lines.extend(self._render_persona(name, states))
        lines.append("╚════════╝")
        return "\n".join(lines)

    def _render_persona(self, name: str, states: List[PersonaState]) -> List[str]:
        first = states[0]
        sym = "◉ working" if first.status == "working" else "○ idle"
        out = [f"║ {name} [{sym}]"]
        if first.task:
            out.append(f"║   {first.task}")
        out.append(f"║   tokens: {self.TOKEN_FORMAT.format(first.tokens_k)}")
        for sub in states[1:]:  # 다중 버전 — F-M3-4 sub-row prefix
            sub_task = sub.task or "(idle)"
            out.append(f"║ {self.SUB_ROW_PREFIX} {sub_task}")
            out.append(f"║   tokens: {self.TOKEN_FORMAT.format(sub.tokens_k)}")
        return out

    def _group_by_name(self, personas: List[PersonaState]) -> Dict[str, List[PersonaState]]:
        out: Dict[str, List[PersonaState]] = {}
        for p in personas:
            out.setdefault(p.name, []).append(p)
        return out
```

```python
# scripts/dashboard/notifier.py

import subprocess
import sys
from abc import ABC, abstractmethod
from datetime import datetime, timedelta
from .models import PendingQuestion


class Notifier(ABC):
    """알림 채널 abstraction — Q3 운영자 결정 = osascript 기본."""

    @abstractmethod
    def notify(self, q: PendingQuestion) -> bool:
        ...


class OSAScriptNotifier(Notifier):
    """macOS 기본 — Pushover / CCNotify 회피 (Q3 운영자 결정).

    R-11 Critical 정정 (옵션 1 본 stage 정정):
        drafter v1 lru_cache 패턴은 메모이제이션 = 항상 캐시된 True 반환 → dedupe 영구 미작동.
        finalizer 정정 = dict[str, datetime] + 5분 TTL 비교 패턴으로 정정.
    """

    DEDUPE_TTL = timedelta(minutes=5)  # R-4/R-11 정정: 5분 TTL 명시

    def __init__(self) -> None:
        # R-11 정정: lru_cache → dict (key → 첫 발화 시각)
        self._sent_keys: dict[str, datetime] = {}

    def _is_recently_sent(self, dedupe_key: str) -> bool:
        """5분 TTL — 동일 dedupe_key 중복 발화 방지 (F-X-7-#3, R-11 정정 후 정상 작동)."""
        now = datetime.now()
        last_sent = self._sent_keys.get(dedupe_key)
        if last_sent and (now - last_sent) < self.DEDUPE_TTL:
            return True  # 최근 5분 내 발화 → dedupe (skip)
        self._sent_keys[dedupe_key] = now
        # LRU 동등 동작: 128개 초과 시 가장 오래된 키 제거 (메모리 leak 회피)
        if len(self._sent_keys) > 128:
            oldest_key = min(self._sent_keys, key=self._sent_keys.get)
            del self._sent_keys[oldest_key]
        return False  # 신규 발화

    def notify(self, q: PendingQuestion) -> bool:
        if self._is_recently_sent(q.dedupe_key()):
            return False  # 중복 — skip
        msg = self._sanitize(q.description)  # Sec.16 보안 — AppleScript injection 회피
        title = "jOneFlow"
        # drafter R-5: sound name "Submarine" 영구 박음 (디자인팀 boundary slot #6 신규 후보)
        cmd = [
            "osascript",
            "-e",
            f'display notification "{msg}" with title "{title}" sound name "Submarine"',
        ]
        try:
            result = subprocess.run(cmd, capture_output=True, timeout=5.0, check=False)
            return result.returncode == 0
        except (subprocess.TimeoutExpired, OSError):
            return False  # Sec.14 에러 경로

    @staticmethod
    def _sanitize(s: str) -> str:
        """AppleScript injection 회피 — `"` / `\\` / 줄바꿈 escape."""
        return s.replace("\\", "\\\\").replace('"', '\\"').replace("\n", " ")


class WindowsNotifier(Notifier):
    """Q4 P1 유지 — skeleton만. v0.6.5+ 본 가동."""

    def notify(self, q: PendingQuestion) -> bool:
        # plyer 또는 win10toast 채택 (drafter 자율 비교 = Sec.10.2)
        # v0.6.4 = stub (운영자 결정 = P1 유지)
        return False


def get_notifier() -> Notifier:

    elif sys.platform == "win32":
        return WindowsNotifier()
    else:
        # Linux / WSL — Q3 fallback skip (Stage 5 이월 처리 영역)
```

**finalizer 정정 (R-11 Critical, 옵션 1 본 stage 정정):** drafter v1 `lru_cache(maxsize=128)` 패턴은 메모이제이션이라 항상 캐시된 `True` 반환 → dedupe 영구 미작동. `dict[str, datetime] + 5분 TTL 비교` 패턴(`_is_recently_sent()`)으로 정정. dedupe_key는 5분 단위 truncate(R-4 정정)와 정합. LRU 동등 동작(128 초과 시 가장 오래된 키 제거) 박음으로 메모리 leak 회피. **Stage 8 진입 시 운영자 알림 폭주 회피 + spec 재해석 부담 0건 + brainstorm 의제 6 알림 정합 보장.**

### 9.4 dedupe TTL 5분 알고리즘 (R-11 정정 후)

`PendingQuestion.dedupe_key()`가 timestamp를 5분 단위로 안정화하므로(Sec.3.2, R-4 정정), 같은 5분 윈도우 내 동일 Q는 동일 키 생성 → `_is_recently_sent()` 측 `dict + 5분 TTL` 비교 패턴(R-11 정정)이 정상 작동합니다. `dedupe_key()` 5분 truncate + `_is_recently_sent()` 5분 TTL 비교 = **이중 5분 보장**으로 dedupe 안정성 +1.

Stage 8에서 `cachetools.TTLCache(maxsize=128, ttl=300)` 교체는 시간 기반 정확 expiry가 추가로 필요한 영역에서 채택 가능(Stage 8 이월 AC-S8-4). 본 stage `dict + TTL 비교` 패턴은 `cachetools` 외부 의존 0건이라 `requirements.txt` 추가 0건 정합 + dedupe 의도 정상 작동 영역.

### 14.1 분류

### 14.2 폴백 체인

```
Q2 토큰량:
  hook 파일 read → JSON parse → 성공 → 반환
                              → 실패 → capture-pane regex 폴백
                                       → 성공 → 반환
                                       → 실패 → 0.0 반환

상태 추론:
  tmux session 존재? → No → idle 폴백 (Q5) — last_update 보존(R-1 정정)
                     → Yes → last_pane_change → idle 임계 T 비교
                                              → working 또는 idle
```

### 14.3 staleness 표시 영역 (R-1 정정 후 정합)

`PersonaState.last_update`가 polling 주기 + 1초(= 2초)보다 오래 stale이면 status 기호 옆에 `⚠`를 표시합니다:

```
║ 박지영 [◉ working ⚠]
║   v0.6.4/Stage 5
║   tokens: 145.2k (stale)
```

**R-1 정정 흡수 정합:** idle 폴백 시 `last_update`가 `datetime.now()`로 갱신되지 않고 이전 값(`_last_known_states`) 보존 → tmux 세션 미존재 페르소나는 자연스럽게 stale 표시 발동. 운영자 가시성 신규 정책 정상 작동 영역.
