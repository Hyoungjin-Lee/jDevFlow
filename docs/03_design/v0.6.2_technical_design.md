## Sec. 0. 본 문서의 위치와 범위

본 design은 v0.6.2의 5개 planning final(01~05)을 단일 trail 기술 설계로 수렴한다. 별도 module-by-module 설계 분리 없이 **모듈 경계 → 데이터 흐름 → 에러 경로 → Stage 5 이월 12건 결정 → 테스트 → 보안 → AC 마감**의 한 줄로 묶는다.

본 design이 결정하는 것:
1. v0.6.2 5개 항목의 모듈 경계 + 인터페이스 합의
2. 5개 항목 사이의 cross-cutting 데이터 흐름 (HANDOFF symlink, settings.json 비결합, R2 진입 순서)
3. 12건 Stage 5 이월 항목 모두 결정 명시
4. v0.6 설계 제약 5건 계승 검증
5. 자동화 hook(ai_step.sh, init_project.sh) 삽입 위치 + 헬퍼 시그니처
6. AC 40건 측정 가능성 마감 (Stage 5 이월 5건 → design 결정 후 자동 검증 가능 상태로 전환)

본 design이 결정하지 않는 것 (Stage 8 구현자 자율):
- 각 파일의 정확한 본문 워딩 (예: operating_manual.md 6개 섹션 내부 구성)
- README 라이선스 뱃지 디자인 (Q2 운영자 결정)
- HR팀 추가 시점 (planning_01 v0.6.2 후속 이월)

## Sec. 2. 모듈 경계 (Module Boundaries)

v0.6.2는 **5개 독립 모듈 + 1개 cross-cutting 헬퍼**로 구성된다. 각 모듈은 자체 ownership 영역과 외부 인터페이스를 가지며, 모듈 간 결합은 cross-cutting 헬퍼와 명시적 file path 참조로만 발생.

### 2.1 모듈 인벤토리

| Module | Ownership 영역 | 외부 인터페이스 | 의존 |
|--------|--------------|---------------|------|
| **M-Org** | `CLAUDE.md` Sec.2.5 (5~10줄 포인터), `docs/operating_manual.md#조직도` (5계층 + 18명 표) | `CLAUDE.md → docs/operating_manual.md#조직도` 1단방향 link | M-SelfEdu (operating_manual.md를 함께 소유) |
| **M-License** | 루트 `LICENSE`, `README.md` 라이선스 섹션, `README.ko.md` 라이선스 섹션, `.py`/`.sh` SPDX 1줄 헤더 | 외부: Apache 2.0 표준 전문 (apache.org). 내부: SPDX-License-Identifier 토큰 | 없음 (cold start) |
| **M-Slash** | `.claude/commands/{init-project,switch-team,ai-step}.md` 3개 markdown | 내부: `bash scripts/{init_project,switch_team,ai_step}.sh` 호출 1줄 | M-Handoffs (`/ai-step --auto` 회귀, F-X-3) |
| **M-Handoffs** | `handoffs/active/`, `handoffs/archive/`, `HANDOFF.md` symlink, `scripts/ai_step.sh` Stage 13 hook, `scripts/init_project.sh` 새 버전 hook | 내부: symlink + frontmatter `status` 필드. 외부: 없음. | M-SelfEdu (cp -r 범위, F-EDU-D3), 헬퍼 `_frontmatter_set_field` |
| **M-SelfEdu** | `CLAUDE.md` 슬림 본체 (~80줄), `docs/operating_manual.md` (≤ 1000줄, 6개 섹션), `scripts/init_project.sh` scaffold cp -r 범위 확장 | R2 순서: `CLAUDE.md → bridge_protocol.md → operating_manual.md → handoffs/active/HANDOFF_v<X>.md` | M-Org (조직도 거주처 공동 소유), M-Handoffs (handoffs/ 구조 설명) |
| **H-Frontmatter** (cross-cutting) | `scripts/lib/settings.sh` `_frontmatter_set_field "<file>" "<key>" "<value>"` POSIX 헬퍼 (sed -i BSD/GNU 동시 호환) | 내부: M-Handoffs hook 두 곳에서 호출 | sed (BSD/GNU 모두) |

### 2.3 모듈 ownership 충돌 해소

**충돌 1 (F-X-1):** `CLAUDE.md` Sec.2.5 ↔ `docs/operating_manual.md#조직도`. **해결:** M-Org가 양쪽 파일을 동시 소유하되, **Sec.2.5는 5~10줄 포인터, operating_manual.md#조직도가 본체**. M-SelfEdu의 ~80줄 슬림화 정책과 자연 양립.

**충돌 2 (F-X-7):** 5개 모듈 구현 순서. **해결:** `M-License → M-Org → M-SelfEdu → M-Handoffs → M-Slash`. M-License는 독립적이라 cold start로 워밍업, M-Org/M-SelfEdu는 강한 결합이라 인접 배치, M-Handoffs hook 후 M-Slash 회귀 테스트 가능.

**충돌 3 (F-X-3):** `/ai-step --auto` (M-Slash) → `ai_step.sh` Stage 13 archive hook (M-Handoffs). **해결:** M-Slash는 단순 래퍼이므로 hook 동작 영향 없음. 단 Stage 9 코드 리뷰에서 `/ai-step --auto` E2E 시나리오로 hook 발화 확인 (Sec.6.3).

## Sec. 3. 데이터 흐름 (Data Flow)

v0.6.2가 도입하는 4개 핵심 데이터 흐름.

### 3.1 HANDOFF symlink 라이프사이클 (M-Handoffs)

```
[init_project.sh 새 버전 시작]
    │
    │ _init_handoff_active "v0.6.2"
    ▼
handoffs/active/HANDOFF_v0.6.2.md   ← 신규 생성, status: active
    ▲
    │ ln -s handoffs/active/HANDOFF_v0.6.2.md HANDOFF.md
    │
HANDOFF.md (symlink)   ← R2 진입점, readlink로 검증

[운영자 작업 진행 — Stage 1~12]
    │
    │ ... 정상 워크플로
    ▼
[ai_step.sh Stage 13 완료]
    │
    │ _ai_step_archive_handoff
    │ 1. _frontmatter_set_field "active_file" status archived
    │ 2. mv handoffs/active/HANDOFF_v0.6.2.md handoffs/archive/
    │ 3. rm -f HANDOFF.md   (옵션 A — 부재, F-04-S5a)
    ▼
handoffs/archive/HANDOFF_v0.6.2.md   ← status: archived (영구 보존)
HANDOFF.md   ← 부재 (다음 init까지)

[다음 init_project.sh 호출 시 새 active 생성으로 cycle 복귀]
```

**불변식:**
- `handoffs/active/`에 `status: active`인 파일은 항상 0개 또는 1개. `preparing` 다수 가능.
- `HANDOFF.md`가 존재하면 반드시 symlink + target은 `handoffs/active/HANDOFF_v*.md`.
- `handoffs/archive/`는 append-only (영구 보존, Q4 finalizer 결정).

### 3.2 slash command 호출 흐름 (M-Slash)

```
사용자 입력: /ai-step --auto
    │
    │ Claude Code 슬래시 명령어 라우터
    ▼
.claude/commands/ai-step.md
    │ frontmatter 검증 (allowed-tools: bash)
    │ 본문 instruction 해석
    ▼
bash scripts/ai_step.sh --auto
    │ (인자 1:1 매핑, escape 책임은 ai_step.sh)
    ▼
ai_step.sh 본체 (v0.6 구현, 미수정)
    │ resolve_executor / check_complete / dispatch
    │ Stage 13 도달 시 → archive hook 발화 (M-Handoffs)
    ▼
정상 종료 또는 approval gate paused
```

**비결합 정책 (F-62-1, F-SLA-3):** `.claude/commands/<cmd>.md` frontmatter는 `.claude/settings.json` 필드를 직접 참조하지 않는다. 모든 동적 결정(team_mode, stage_assignments)은 `bash scripts/<sh>` 내부에서 수행. 래퍼는 인자 형식만 strict하게 정의.

### 3.3 R2 진입 순서 (M-SelfEdu, F-X-5)

새 사용자(또는 백지 Claude)의 jOneFlow 학습 진입 순서를 단일 그래프로 고정한다.

```
1. CLAUDE.md          (~80줄, 절대 규칙 + 포인터)
        ▼
2. bridge_protocol.md (회의창 ↔ 브릿지 ↔ 오케 통신 모델)
        ▼
3. docs/operating_manual.md  (≤ 1000줄, 6개 섹션 완전 자족 매뉴얼)
        ▼
4. handoffs/active/HANDOFF_v<X>.md  (현재 진행 상태)
```

**검증:** AC-Edu-7 측정 명령 — `grep -c "operating_manual" CLAUDE.md ≥ 1` + `grep -c "R2.*순서\|읽기 순서" docs/operating_manual.md ≥ 1`.

### 3.4 settings.json 보존 흐름 (F-X-4 / F-D3)

v0.6.2는 **`.claude/settings.json` schema를 변경하지 않는다.** v0.4 schema 그대로 보존.

```
v0.6 schema v0.4 (현재)            v0.6.2 (변경 없음)
─────────────────────              ──────────────────
schema_version: "0.4"          →   schema_version: "0.4"
workflow_mode: ...             →   workflow_mode: ...
team_mode: ...                 →   team_mode: ...
stage_assignments: ...         →   stage_assignments: ...

(personas 필드 추가 안 함)         (F-X-4 결정: 추가 안 함)
(pending_team_mode 부재 유지)      (F-D3 보존)
```

**근거:** 18명 페르소나는 운영 매뉴얼 영역(`docs/operating_manual.md#조직도`)에 텍스트로만 정의. 실행 시점 매핑은 v0.4 schema의 `stage_assignments` 4개 키(`stage8_impl`/`stage9_review`/`stage10_fix`/`stage11_verify`)로 충분. 페르소나 11명/18명 변동은 settings.json 형식과 무관.

## Sec. 4. 에러 경로 (Error Paths)

5개 모듈 주요 에러 시나리오 + fail-closed 정책.

| Path ID | 시나리오 | 발화 위치 | 처리 정책 |
|---------|---------|---------|----------|
| **EP-1** | symlink 부재 (HANDOFF.md 없음, 다음 init 전) | R2 진입 시 또는 grep | 옵션 A 정책: 부재가 정상 상태. 운영자에게 `init_project.sh` 호출 안내. M-Handoffs hook의 `rm -f` 직후 의도된 부재. |
| **EP-2** | symlink target 누락 (handoffs/active/HANDOFF_v*.md 삭제됨) | `cat HANDOFF.md` 실패 | fail-closed. `readlink HANDOFF.md`로 target 확인 후 `[ -f "$(readlink HANDOFF.md)" ]` guard. 누락 발견 시 `init_project.sh` 또는 마지막 archive에서 복구 안내. |
| **EP-3** | Windows symlink 미지원 (NTFS / git core.symlinks=false) | git clone 직후 | F-04-S5b 정책: WSL 사용 권장. `.gitattributes`에 `HANDOFF.md symlink` 명시. Linux/macOS만 1차 지원. Windows는 fallback 가이드 (`docs/operating_manual.md` 별도 섹션). |
| **EP-4** | sed -i 호환성 깨짐 (BSD vs GNU) | `_frontmatter_set_field` 호출 | F-04-3 헬퍼: BSD(`-i ''`) / GNU(`-i`) 분기는 헬퍼 내부에서 `sed --version 2>/dev/null \| grep -q GNU` 감지 후 대응. awk fallback 사용. |

**시그니처:**
```sh
# Usage: _frontmatter_set_field <file> <key> <value>
# Effect: <file> frontmatter (--- ... ---) 안의 <key>: <value> 갱신.
#         키 부재 시 frontmatter 끝부분에 추가.
#         BSD sed와 GNU sed 모두 호환 (-i '' vs -i 분기).
# Returns: 0 (성공), 1 (frontmatter 부재), 2 (인자 부족).
_frontmatter_set_field() {
    local _file="$1" _key="$2" _value="$3"
    [ "$#" -eq 3 ] || return 2
    [ -f "$_file" ] || return 2
    head -1 "$_file" | grep -q '^---$' || return 1
    
    # awk 기반 (sed -i 호환성 회피)
    local _tmp="$_file.tmp.$$"
    awk -v key="$_key" -v val="$_value" '
        BEGIN { in_fm=0; done=0 }
        /^---$/ { 
            if (in_fm == 0) { in_fm=1; print; next }
            else if (in_fm == 1 && done == 0) { print key ": " val; done=1; print; next }
            else { print; next }
        }
        in_fm == 1 && /^[a-zA-Z_][a-zA-Z0-9_-]*: / {
            split($0, parts, ":")
            if (parts[1] == key) { print key ": " val; done=1; next }
        }
        { print }
    ' "$_file" > "$_tmp" && mv "$_tmp" "$_file"
}
```

**근거 (F-D2 준수):** awk 기반으로 sed -i BSD/GNU 분기 회피. POSIX awk만 사용. jq 비의존.

### 7.1 LICENSE 변경 영향 (M-License)

- **외부 영향:** 기존 MIT 사용자/외부 fork에 Apache 2.0 전환 공지 필요. Q6(외부 fork 확인) 미완료 시 위험. **Stage 4.5 게이트 통과 후 진행 가정.**

### 7.2 .claude/commands/ frontmatter 비결합 (M-Slash, F-62-1)

- **위험 차단:** frontmatter에서 settings.json 동적 참조 가능했다면 → 사용자 입력으로 settings.json 임의 키 변경 attack vector. 본 design은 단순 bash 호출만 허용 → 위험 0.

### 7.3 HANDOFF.md symlink 안전성 (M-Handoffs, F-62-2)

- **target 검증:** symlink는 임의 path를 가리킬 수 있음. R2 진입 시 `readlink HANDOFF.md` 출력이 `handoffs/active/HANDOFF_v` prefix로 시작하는지 가드.
- **archive 후 부재 (EP-1):** rm -f 직후 부재는 의도된 상태. 외부 자동화가 부재를 에러로 오인하지 않도록 EP-1 정상 케이스 명시.
- **git symlink 추적:** `.gitattributes`에 명시 → clone 시 symlink 복원 보장 (Linux/macOS만, Windows는 EP-3).
