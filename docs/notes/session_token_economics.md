# 💰 세션 토큰 경제학 — "지금 이어갈까, 새 세션으로 넘길까"

> **목적:** 실제 프로젝트 운영 중 "지금 세션에서 계속할지, 새 세션으로 넘길지" 판단이 필요할 때 참고할 운영 가이드.
> **성격:** 언어 정책과 무관한 **운영 지혜 / 내부 참조**. Stage 산출물 아님. (EN 페어는 OSS 공개 시점에 추가 예정.)
> **최초 작성:** 2026-04-22 (jOneFlow v0.3 세션 2 중 실제 판단 사례에서 유도)
> **유지 정책:** 새로운 사례·오해·반례가 쌓이면 하단 "§7. 사례 로그"에 추가.

---

## 1. 이 문서가 필요한 이유

Claude 세션을 길게 운영하다 보면 반드시 마주치는 질문이 있다:

> **"이 작업 지금 이 세션에서 계속하는 게 싸, 아니면 새 세션으로 넘기는 게 싸?"**

직관만으론 틀리기 쉽고, 한 번 추론했다가 다음에 같은 고민을 하면서 또 처음부터 생각하게 된다. 그래서 이 문서가 있다.

**결론 요약** (먼저 보고 싶은 사람용):

1. **"세션 사용량 %"와 "토큰 비용"은 다른 축이다.** 새 세션 = 예산 리셋이 아니다.
2. **Input 토큰은 매 턴마다 이력 전체가 재전송된다.** 따라서 턴 수가 많을수록 input 비용이 복리로 불어난다.
3. **Prompt caching**이 이 복리 효과를 많이 상쇄해주지만, **새 세션에서는 워밍업부터 다시 해야** 한다.
4. **판단 규칙** (대략):
   - 1~2턴이면 끝 → **지금 세션 유지**
   - 3~5턴 중간 규모 → **경우에 따라 다름, 보통 유지가 약간 유리**
   - 6턴 이상 반복 작업 → **새 세션이 유리**
   - **사용량이 70%+ 도달** → 토큰보다 **중간 아웃 리스크**가 크니 새 세션 전환

## 2. 핵심 개념 — 헷갈리기 쉬운 두 축

| 축 | 무엇인가 | 어떻게 영향받는가 |
|----|---------|----------------|
| **세션 사용량 %** (rate limit) | 계정 단위 5시간 윈도우 한도 | 새 세션으로 바꿔도 **리셋 안 됨**. 같은 5시간 창 안에서는 같은 예산에서 빼감. |
| **실제 토큰 소비량** | API 과금 또는 rate limit 소진의 **정량적 근원** | 어느 경로가 **실제로 더 적은 토큰을 쓰느냐**는 전혀 다른 판단. |

> **핵심 오해:** "새 세션 가면 예산이 회복된다"는 생각은 틀렸다. 새 세션이 유리할 수 있는 이유는 **그 쪽이 총 토큰을 덜 먹기 때문**이지, 예산이 리셋되기 때문이 아니다.

---

## 3. Input 토큰의 복리 효과

LLM 과금의 핵심 메커니즘:

> **매 턴마다, 직전까지의 모든 대화 이력이 input으로 재전송된다.**

결과적으로:

- **Output 토큰**은 그 턴에 생성된 만큼만 (한 번)
- **Input 토큰**은 매 턴마다 과거 전체를 다시 낸다 (누적)

### 단순 모델

세션에서 N턴 진행 시 대략적 input 총량:

```
Input_total ≈ Σ (시스템 프롬프트 + Σ_{i<t} (이전 턴 input + output))
            = O(N²) 성장  (캐시 없을 때)
```

즉 **턴 수가 두 배 되면 input 비용은 네 배 가까이** 된다 (캐시 없이는).

### Prompt caching의 역할

Anthropic은 **prompt caching**을 제공한다:

- 반복되는 앞부분 컨텍스트를 캐시로 표시하면, **캐시 히트 시 90% 할인**
- TTL: 기본 5분, 확장 1시간
- **세션 범위**: 같은 세션 안에서 효과 최대
- **세션 간**: 캐시 공유 없음 (새 세션은 워밍업부터)

캐시가 잘 듣는 조건에서는 O(N²)이 거의 O(N)으로 떨어진다. 그래서 **활성 세션 유지 시 토큰 비용 이점**이 큰 것.

### 새 세션의 트레이드오프

| 항목 | 현재 세션 유지 | 새 세션 전환 |
|------|---------------|-------------|
| 캐시 활용 | 최대 (기존 워밍업 상태 이용) | 0 → 재워밍업 필요 |
| Baseline input 크기 | 크다 (누적된 이력) | 작다 (새 출발) |
| 턴당 비용 증가율 | 큰 baseline × 캐시 할인 | 작은 baseline × 캐시 할인 |
| 재로드 세금 | 없음 | HANDOFF + 관련 문서 재읽기 (~15~25K 토큰) |
| 중간 아웃 리스크 | 누적된 모든 작업 날아감 | 작업 초기라 손실 적음 |

## 4. 손익분기점 Heuristic

경험적으로:

| 남은 작업량 | 유리한 쪽 | 이유 |
|------------|----------|------|
| **1~2턴 집중 완료** | 현재 세션 계속 | 재로드 오버헤드 회피 + 캐시 최대 활용 |
| **3~5턴 중간 규모** | 경계 영역 — 보통 현재 세션이 약간 유리 | 캐시가 baseline 크기를 상쇄 |
| **6턴+ 반복/이터레이션** | 새 세션 유리 | Input 복리가 캐시 할인을 이김 |
| **사용량 70%+ 도달** | 새 세션 **강하게** 유리 | 토큰 비용보다 **중간 아웃 리스크**가 훨씬 비쌈 |
| **역할 전환 필요** (drafter→reviewer 등) | 새 세션 | confirmation bias 방지 부수 효과 |

### 실무 의사결정 플로 (간단판)

```
사용량이 70% 이상인가?
├─ YES → 새 세션으로 (리스크 회피 우선)
└─ NO
   └─ 남은 작업이 1~2턴 안에 끝나는가?
      ├─ YES → 현재 세션 유지
      └─ NO
         └─ 역할 전환이 필요한가 (작성→독립 검증 등)?
            ├─ YES → 새 세션 (토큰 + bias 방지 동시 이득)
            └─ NO → 남은 턴 추정
               ├─ ~5턴 → 현재 유지
               └─ 6턴+ → 새 세션
```

## 5. 이번 판단의 실제 경로 (Worked Example)

**상황** (2026-04-22, jOneFlow v0.3 세션 2 중):

- 누적된 작업: Stage 1 brainstorm 읽기 + Stage 2 plan_draft EN/KO 작성 완료
- 세션 사용량: 시작 시점 76%
- 리셋까지: 약 2시간
- 남은 작업: Stage 3 (review) + Stage 4 (final) + 각각 KO 번역 = 대략 4~6턴

**세션 유지 시 예상 비용**:
- 누적 컨텍스트 ~40~50K (캐시됨)
- 4~6턴 동안 매 턴 input ≈ 40K→55K→70K... (캐시 90% 할인 적용)
- Output ~15~20K

**새 세션 시 예상 비용**:
- 재로드 ~15~20K (CLAUDE + HANDOFF + WORKFLOW + plan_draft)
- 캐시 워밍업 비용 (첫 호출만)
- 4~6턴 동안 매 턴 input ≈ 20K→35K→50K... (캐시 적용)
- Output ~15~20K (동일)

**순수 토큰만 보면**: 현재 세션 계속이 약간 더 저렴할 가능성.

**그러나 결정 요인은 순수 토큰이 아니었다**:
1. **중간 아웃 리스크** — 76% → 중간에 한도 도달하면 이번 세션 작업 전체가 날아갈 비용 > 재로드 세금.
2. **Stage 3 셀프 리뷰 품질** — 방금 draft 쓴 관점에서 리뷰 vs. "drafter의 텍스트를 받아든 리뷰어" 관점의 차이는 confirmation bias 회피 측면에서 후자가 낫다.
3. **HANDOFF.md가 이미 잘 정비되어 있음** — 전환 세금이 상대적으로 낮은 상태.

**결정**: 새 세션 전환. 3가지 우선순위 중 (1) 리스크 회피가 지배적이었고, (2) (3)이 같은 방향을 가리켰다.

**만약 사용량이 30%였다면?** → 반대로 현재 세션 유지가 더 나았을 것. 같은 작업량이어도 리스크 계수가 작으면 재로드 세금을 굳이 낼 이유가 없다.

## 6. 실무 체크리스트

세션 전환 여부 판단할 때 확인할 질문:

### 6-1. 현재 세션 유지가 맞는가 체크

- [ ] 사용량 50% 이하인가?
- [ ] 남은 작업이 2턴 이내에 끝날 수 있는가?
- [ ] 앞 컨텍스트를 자주 참조해야 하는가 (복잡한 상호 의존)?
- [ ] 역할 전환이 필요 없는가 (계속 같은 역할)?
- [ ] "이 세션 끝나기 전에 이건 꼭 마친다"는 명확한 out 조건이 있는가?

**3개 이상 YES** → 현재 세션 유지.

### 6-2. 새 세션 전환이 맞는가 체크

- [ ] 사용량 70% 이상인가?
- [ ] 남은 작업이 길고 이터레이션이 필요한가?
- [ ] Stage 11 같은 **독립 검증**이 필요한 단계인가?
- [ ] HANDOFF.md가 잘 정비되어 있는가 (전환 세금 낮음)?
- [ ] 작성의 감정적 관성을 끊고 싶은가 (바이어스 의심)?

**2개 이상 YES** → 새 세션 전환.

### 6-3. 세션 전환 비용을 낮추는 투자

**HANDOFF.md 품질이 곧 전환 자유도다.**

jOneFlow가 HANDOFF.md를 강조하는 이유가 바로 이것: 세션 경계를 값싸게 넘을 수 있게 하는 **사전 투자**. 매 세션 종료 시 HANDOFF를 잘 업데이트하면, 다음에 세션 전환 판단할 때 "전환 비용" 항목이 거의 0에 가까워진다.

**투자 포인트**:
- HANDOFF.md의 "Next Session Prompt" — 복사-붙여넣기로 맥락 복구 가능한 수준
- 주요 결정(D-A, D-B, verdict 등)은 문서에 박아둠 — 채팅 이력에 의존 금지
- 현재 Stage의 focus / 다음 Stage의 입구 조건 명시
